# Sprint 5, Story 9.3: Dev Auto-Deploy — How the GitOps Loop Works

## Story Summary
Understand and verify the automatic deployment loop for the dev environment.
When code is merged to `main`, the CI pipeline builds the image, updates the GitOps repo,
and ArgoCD automatically syncs the cluster. No manual `kubectl apply` — ever.

The full workflow is already implemented in `techitfactory-app/.github/workflows/ci.yml`.

## Part 1: The Complete Dev Deploy Flow

```
Developer pushes commit to main
(touching services/api-gateway/**)
              ↓
detect-changes job
  → dorny/paths-filter identifies api-gateway changed
  → builds matrix: [{"service":"api-gateway","path":"services/api-gateway","type":"node"}]
              ↓
test job (parallel matrix, only for changed services)
  → npm ci → npm run lint → npm test
  → Python: pip install → flake8 → pytest
  → FAILS HERE if tests fail — nothing builds
              ↓
build job (only on push to main, not PRs)
  → OIDC → assume techitfactory-github-terraform role
  → ECR login
  → docker build + push (commit SHA tag + :latest)
  → Trivy scan (CRITICAL/HIGH fail the build)
  → Fails if any CVE found with a fix available
              ↓
update-gitops job (max-parallel: 1)
  → checkout techitfactory-gitops (via GITOPS_TOKEN PAT)
  → yq -i '.images[0].newTag = strenv(IMAGE_TAG)' environments/dev/api-gateway/kustomization.yaml
  → git pull --rebase (absorbs concurrent updates from other services)
  → git commit + push
              ↓
ArgoCD polls techitfactory-gitops every ~3 minutes
  → detects kustomization.yaml changed
  → kustomize build environments/dev/api-gateway
  → kubectl apply the rendered manifests
  → deployment rolls out (rolling update)
              ↓
New pod running with the new image SHA
```

## Part 2: Why yq — Not sed

The `update-gitops` job uses `yq` to update the image tag:

```bash
# What we do (safe):
yq -i '.images[0].newTag = strenv(IMAGE_TAG)' kustomization.yaml

# What the old version did (dangerous):
sed -i "s|newTag:.*|newTag: $SHA|" kustomization.yaml
```

**Why `sed` is wrong:** It matches ANY line containing `newTag` — including comments,
other fields, or nested YAML keys. If your kustomization ever has a second `newTag`
in a comment like `# previous newTag: abc123`, `sed` corrupts the file silently.
`yq` navigates the YAML structure and only updates `.images[0].newTag`.

---

## Part 3: Why max-parallel: 1 on update-gitops

If you push a commit that touches both `api-gateway` and `product-service`, CI builds
them in parallel — but then BOTH try to `git push` to techitfactory-gitops at the same time.
The second push would be rejected (non-fast-forward) or overwrite the first update.

`max-parallel: 1` forces the matrix jobs to run one at a time in the `update-gitops` step.
Combined with `git pull --rebase`, this serializes GitOps commits cleanly:

```
api-gateway update → git pull --rebase → commit → push → SUCCESS
                                                              ↓
product-service update → git pull --rebase → commit → push → SUCCESS
```

## Part 4: Concurrency Control

```yaml
concurrency:
  group: ci-${{ github.ref }}
  cancel-in-progress: true
```

If you push two commits in quick succession:
- Run 1 starts (detect → test → build)
- Run 2 starts → Run 1 is **cancelled**
- Run 2 finishes with the latest commit SHA

This is safe because:
- ECR tags are content-addressable (same SHA = same image)
- GitOps ends up with the latest commit SHA — which is what you want
- Avoids two stale runs racing to update GitOps with an older SHA

## Part 5: Verify the Dev Deploy End-to-End

### Step 1: Trigger a CI Run
```bash
cd ~/Desktop/Devops-Project/techitfactory-app

# Touch one service to trigger CI for only that service
echo "# ci test $(date)" >> services/api-gateway/README.md
git add services/api-gateway/README.md
git commit -m "test: trigger CI for api-gateway"
git push
```

### Step 2: Watch the Pipeline
GitHub → `techitfactory-app` → **Actions** tab

You should see 4 jobs:
1. `Detect Changes` → shows `api-gateway: true`
2. `Test api-gateway` → lint + tests pass
3. `Build api-gateway` → Docker build + ECR push + Trivy scan
4. `Update GitOps` → kustomization.yaml commit

### Step 3: Verify ECR Image Was Pushed
```bash
aws ecr describe-images \
  --repository-name techitfactory/api-gateway \
  --profile techitfactory \
  --region ap-south-1 \
  --query 'imageDetails | sort_by(@, &imagePushedAt) | [-1].imageTags' \
  --output table
```

### Step 4: Verify GitOps Was Updated
```bash
cd ~/Desktop/Devops-Project/techitfactory-gitops
git pull
grep newTag environments/dev/api-gateway/kustomization.yaml
# Should show a 40-char git SHA
```

### Step 5: Verify ArgoCD Synced
```bash
argocd app get api-gateway --refresh
# Status: Synced, Healthy
# Image tag should match the SHA from step 4
```
If ArgoCD hasn't synced yet (< 3 mins), force it:
```bash
argocd app sync api-gateway
```

### Step 6: Verify Pod Is Running the New Image
```bash
kubectl get pods -n techitfactory -l app=api-gateway
kubectl describe pod <pod-name> -n techitfactory | grep Image:
# Should show: ...ecr...amazonaws.com/techitfactory/api-gateway:<sha>
```

## Part 6: ArgoCD Auto-Sync Settings (Verify)

Dev apps should have auto-sync enabled. Check it:

```bash
argocd app get api-gateway -o json | jq '.spec.syncPolicy'
```

Expected output:
```json
{
  "automated": {
    "prune": true,
    "selfHeal": true
  }
}
```

If not set, patch it:
```bash
argocd app set api-gateway --sync-policy automated --auto-prune --self-heal
```

- **`prune: true`** — resources removed from GitOps are deleted from the cluster
- **`selfHeal: true`** — if someone manually does `kubectl apply`, ArgoCD reverts it back to Git state within 3 minutes

## Part 7: Rollback

### Option A — ArgoCD CLI (fastest)
```bash
# Roll back to the previous sync
argocd app rollback api-gateway

# Roll back to a specific history entry
argocd app history api-gateway
argocd app rollback api-gateway <history-id>
```

### Option B — GitOps Revert (recommended — keeps git as source of truth)
```bash
cd ~/Desktop/Devops-Project/techitfactory-gitops
git log environments/dev/api-gateway/
git revert HEAD
git push
# ArgoCD auto-syncs → old image tag is deployed
```

### Option C — kubectl Direct (emergency only)
```bash
kubectl rollout undo deployment/api-gateway -n techitfactory
# WARNING: This puts git out of sync. Run Option B immediately after to re-align.
```
