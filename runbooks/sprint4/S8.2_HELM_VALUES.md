## Story Summary

different resource needs, probe paths, ports, and scaling behaviour.

- HPA (autoscaling) enabled for high-traffic services

## Part 1: How the Helm Chart Works

The project uses a **single shared Helm chart** with per-service values files:

```
charts/microservice/
├── Chart.yaml
├── values.yaml                  ← defaults (all services inherit these)
├── templates/
│   ├── deployment.yaml
│   ├── service.yaml
│   ├── hpa.yaml
│   ├── networkpolicy.yaml
│   └── poddisruptionbudget.yaml
└── values/
    ├── api-gateway.yaml         ← overrides for api-gateway only
    ├── frontend.yaml
    ├── user-service.yaml
    ├── product-service.yaml
    ├── order-service.yaml
    └── cart-service.yaml
```

To deploy a service with Helm manually (learning only — ArgoCD handles this in prod):
```bash
helm upgrade --install api-gateway charts/microservice \
  -f charts/microservice/values/api-gateway.yaml \
  -n techitfactory
```

## Part 2: Resource Sizing Per Service

Different services need different resources. Here's the sizing rationale:

| Service | CPU Request | CPU Limit | Mem Request | Mem Limit | Why |
|---------|------------|-----------|-------------|-----------|-----|
| frontend | 50m | 200m | 64Mi | 128Mi | nginx static — very low |
| api-gateway | 100m | 500m | 128Mi | 256Mi | proxies all traffic |
| user-service | 100m | 300m | 128Mi | 256Mi | JWT + bcrypt = CPU heavy |
| product-service | 50m | 200m | 128Mi | 256Mi | read-only mock data |
| order-service | 100m | 300m | 128Mi | 256Mi | Python — higher base memory |
| cart-service | 50m | 200m | 128Mi | 256Mi | in-memory, lightweight |

Check current resource requests vs actual usage:
```bash
kubectl top pods -n techitfactory
```

## Part 3: Enable HPA for High-Traffic Services

API Gateway and User Service handle the most load — enable autoscaling:

```bash
# Check current HPA status
kubectl get hpa -n techitfactory
```

The HPA is controlled by `autoscaling.enabled` in values. For api-gateway:

```yaml
# charts/microservice/values/api-gateway.yaml
autoscaling:
  enabled: true
  minReplicas: 2
  maxReplicas: 6
  targetCPUUtilizationPercentage: 70
```

After updating the values file, the CI/CD pipeline picks it up automatically.
For a manual test:
```bash
# Trigger HPA manually to verify it works
kubectl autoscale deployment api-gateway \
  --cpu-percent=70 --min=2 --max=6 \
  -n techitfactory

kubectl get hpa -n techitfactory -w
```

## Part 4: Health Probe Paths Per Service

Every service exposes `/health` (liveness) and `/ready` (readiness). Verify
these match what the actual service code returns:

```bash
# Test each service's health probe path
for svc in api-gateway user-service product-service cart-service; do
  echo -n "${svc}: "
  kubectl exec -it deploy/api-gateway -n techitfactory -- \
    wget -qO- http://${svc}/health 2>/dev/null || echo "FAILED"
done

# Order service runs on port 8000
echo -n "order-service: "
kubectl exec -it deploy/api-gateway -n techitfactory -- \
  wget -qO- http://order-service:8000/health 2>/dev/null
```

> **Note:** Order service is Python/FastAPI on port 8000. All Node.js services
> are on port 3000. The startup probe gives each service a 90-second window
> (30 × 5s) before liveness kicks in.
