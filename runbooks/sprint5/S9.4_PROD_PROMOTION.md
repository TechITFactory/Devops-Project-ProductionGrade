# Sprint 5, Story 9.4: Prod Promotion via GitHub Release

## Story Summary
Production deployments are triggered by publishing a GitHub Release with a semver tag.
A mandatory approval gate blocks prod GitOps updates until a reviewer signs off.
ArgoCD prod apps use **manual sync** — the cluster change requires a separate human action.

The full workflow is already implemented at `techitfactory-app/.github/workflows/release.yml`.

---

## What Was Fixed From the Original

| Old | Fixed |
|---|---|
| No Trivy scan on prod builds | Trivy now runs on every service in `build-all-services` |
| `:latest` tag pushed to prod ECR | Removed — prod images only get `vX.Y.Z` tag |
| `yq -i ... env(RELEASE_TAG)` — env() deprecated | Use `strenv(RELEASE_TAG)` — correct yq v4 syntax |
| No `git pull --rebase` before push | Added — prevents push rejection if GitOps has unrelated commits |
| `slackapi/slack-github-action@v1` — unpinned | Pinned to `@v1.27.0` |
| No `concurrency:` | Added `cancel-in-progress: false` — never cancel a release build |
| Slack only on success | `if: always()` — notifies even when approval is denied or build fails |

## Part 1: How the Release Flow Works

```
gh release create v1.0.0
         ↓
release.yml triggered (on: release: types: [published])
         ↓
validate-release
  → checks tag matches ^v[0-9]+\.[0-9]+\.[0-9]+$
  → exits 1 if someone creates a release like "v1.0" or "1.0.0"
         ↓
build-all-services (matrix: all 6 services, fail-fast: false)
  → OIDC → ECR login
  → docker build + push  (tag: vX.Y.Z  — NO :latest for prod)
  → Trivy scan (CRITICAL/HIGH blocks the release — same standard as dev)
  → All 6 services run in parallel
         ↓
update-prod-gitops
  → 🔒 PAUSES — GitHub "production" environment approval required
  → Team lead reviews and approves (or denies)
         ↓  (after approval)
  → checkout techitfactory-gitops (GITOPS_TOKEN PAT)
  → yq -i '.images[0].newTag = strenv(RELEASE_TAG)' for each service
  → git pull --rebase → commit → push
         ↓
notify-release (if: always())
  → Slack message: success or failure
         ↓
ArgoCD prod apps detect GitOps change
  → prod apps are set to MANUAL sync — does NOT auto-deploy
  → Team lead runs: argocd app sync <service>-prod
         ↓
New version live in production
```

## Part 2: Why No `:latest` Tag on Prod Builds

Dev CI pushes two tags per image:
- `<commit-sha>` — the specific build
- `:latest` — points to the most recent dev build

Release workflow pushes **only** `vX.Y.Z`. No `:latest`.

**Why:** If prod images also updated `:latest`, it becomes unclear which image is running
where. ECR's `:latest` would sometimes point to a dev build, sometimes a release.
Use explicit version tags in prod — `v1.2.3` always means `v1.2.3`, nothing else.

## Part 3: Configure GitHub Environment (One-Time Setup)

### Create `production` Environment in techitfactory-app
1. GitHub → `techitfactory-app` → **Settings → Environments → New environment**
2. Name: `production`
3. Configure protection rules:
   - ✓ **Required reviewers** — add team leads (at least 1 required)
   - ✓ **Wait timer**: 5 minutes (gives time to cancel if triggered by mistake)
   - ✓ **Deployment branches**: select **"Selected branches and tags"** → `v*` (tags only)
4. Save

### Create `production-infra` Environment in techitfactory-infra
1. Same steps in `techitfactory-infra` repo
2. Name: `production-infra`
3. Same protection rules as above
4. This gates the Terraform apply for prod infrastructure

### What the Approval UI Looks Like
When `build-all-services` finishes, the workflow pauses:
```
update-prod-gitops is waiting for your review
Environment: production
Required reviewers: @teamlead1, @teamlead2
```
The reviewer sees the built images, the release tag, and can approve or deny.
If denied — the prod GitOps update is skipped entirely. ECR images are built but not deployed.

## Part 4: Create Prod ArgoCD Apps (One-Time Setup)

Prod apps are separate from dev apps and use **manual sync**.

### Create ArgoCD Project for Production
```bash
cat <<EOF | kubectl apply -f -
apiVersion: argoproj.io/v1alpha1
kind: AppProject
metadata:
  name: production
  namespace: argocd
spec:
  description: Production workloads
  sourceRepos:
    - 'https://github.com/TechITFactory/techitfactory-gitops.git'
  destinations:
    - namespace: techitfactory-prod
      server: https://kubernetes.default.svc
  clusterResourceWhitelist:
    - group: ''
      kind: Namespace
EOF
```

### Create Prod Application for Each Service

```bash
# Run this for each service (api-gateway, product-service, order-service, cart-service, user-service, frontend)
SERVICE=api-gateway

argocd app create ${SERVICE}-prod \
  --project production \
  --repo https://github.com/TechITFactory/techitfactory-gitops.git \
  --path environments/prod/${SERVICE} \
  --dest-server https://kubernetes.default.svc \
  --dest-namespace techitfactory-prod \
  --sync-option CreateNamespace=true
  # NO --sync-policy automated — manual sync only for prod
```

### Verify Prod Apps Created
```bash
argocd app list | grep prod
# Expected: 6 apps, all showing "OutOfSync" (until first sync)
```

## Part 5: Create Prod Overlays in GitOps Repo

If `environments/prod/` doesn't exist yet:

```bash
cd ~/Desktop/Devops-Project/techitfactory-gitops/environments

for svc in api-gateway product-service order-service cart-service user-service frontend; do
  mkdir -p prod/${svc}
  # Copy dev overlay and update namespace + replica count
  cp dev/${svc}/kustomization.yaml prod/${svc}/kustomization.yaml
  # Update namespace to prod
  sed -i 's/namespace: techitfactory$/namespace: techitfactory-prod/' prod/${svc}/kustomization.yaml
done

git add .
git commit -m "feat: Add prod environment overlays"
git push
```

### Prod Overlay Example (api-gateway)
```yaml
# environments/prod/api-gateway/kustomization.yaml
apiVersion: kustomize.config.k8s.io/v1beta1
kind: Kustomization

namespace: techitfactory-prod

resources:
  - ../../../base/api-gateway

images:
  - name: IMAGE_PLACEHOLDER
    newName: 535002890483.dkr.ecr.ap-south-1.amazonaws.com/techitfactory/api-gateway
    newTag: v1.0.0   # Release workflow updates this field via yq

replicas:
  - name: api-gateway
    count: 3         # 3 replicas in prod vs 1 in dev

patches:
  - patch: |-
      - op: replace
        path: /spec/template/spec/containers/0/resources/requests/cpu
        value: 200m
      - op: replace
        path: /spec/template/spec/containers/0/resources/limits/cpu
        value: 1000m
    target:
      kind: Deployment
      name: api-gateway
```

## Part 6: Create a Production Release

### Via GitHub CLI (Recommended)
```bash
cd ~/Desktop/Devops-Project/techitfactory-app

# Tag the current main commit
git tag v1.0.0
git push origin v1.0.0

# Create the GitHub Release (triggers release.yml)
gh release create v1.0.0 \
  --title "Release v1.0.0" \
  --notes "First production release. All 6 services." \
  --latest
```

### Via GitHub UI
1. `techitfactory-app` → **Releases → Draft a new release**
2. Tag: `v1.0.0` (create on `main`)
3. Title: `Release v1.0.0`
4. Write release notes
5. **Publish release** (NOT "Save as draft" — that doesn't trigger the workflow)

## Part 7: After Approval — Deploy to Production

After the reviewer approves and `update-prod-gitops` completes:

```bash
# Verify prod GitOps was updated
cd ~/Desktop/Devops-Project/techitfactory-gitops
git pull
grep newTag environments/prod/*/kustomization.yaml
# All services should show: newTag: v1.0.0

# Sync each service to production (one at a time — check health between each)
for svc in api-gateway product-service order-service cart-service user-service frontend; do
  echo "Syncing ${svc}-prod..."
  argocd app sync ${svc}-prod
  argocd app wait ${svc}-prod --health --timeout 120
  echo "${svc}-prod is healthy"
done

# Verify prod pods are running the release image
kubectl get pods -n techitfactory-prod
kubectl describe pod -n techitfactory-prod -l app=api-gateway | grep Image:
# Should show: ...amazonaws.com/techitfactory/api-gateway:v1.0.0
```

## Part 8: Prod Rollback

### Option A — Re-run Previous Release (safest, re-triggers approval gate)
```bash
# Roll back to v0.9.0
gh release create v0.9.1 --title "Rollback to v0.9.0" --notes "Rolling back due to issue"
# Then sync ArgoCD to the new release tag
```

### Option B — GitOps Revert (fast)
```bash
cd ~/Desktop/Devops-Project/techitfactory-gitops
git log environments/prod/api-gateway/
git revert HEAD   # or specific commit
git push
# Then manually sync in ArgoCD:
argocd app sync api-gateway-prod
```

### Option C — ArgoCD History Rollback
```bash
argocd app history api-gateway-prod
argocd app rollback api-gateway-prod <history-id>
```

> **Important:** After any Option B or C rollback, push a new GitOps commit to align
> the Git state with what's running — ArgoCD shows "OutOfSync" otherwise.
