# Sprint 5: Complete Execution Guide
# CI/CD — Infrastructure + Application + GitOps

## What This Sprint Delivers

After completing these steps you will have:

- **Infra CI/CD**: Every PR to `techitfactory-infra` runs `terraform plan`. Every merge runs `terraform apply`. No manual Terraform ever again.
- **Platform GitOps**: ArgoCD manages NGINX Ingress, Cluster Autoscaler, and Metrics Server on the cluster. Upgrade = change a version in git.
- **App CI/CD**: Push code → tests run → Docker image built + Trivy scanned → pushed to ECR → kustomization.yaml updated → ArgoCD deploys. Zero manual `kubectl apply`.
- **Production Promotion**: GitHub Release with a semver tag (`v1.0.0`) triggers a gated prod build. A reviewer must approve before prod GitOps is updated.

## Ingress: NGINX + NLB (not ALB)

ALB is restricted in this AWS account. We use **NGINX Ingress Controller** instead:
- Installed by ArgoCD from `apps/platform/nginx-ingress.yaml`
- Creates an AWS **Network Load Balancer** (NLB) via the EKS legacy cloud controller annotation
- No AWS Load Balancer Controller required
- All Ingress resources use `ingressClassName: nginx`

## Pipeline Trigger: `[deploy]` Keyword

Pipelines do **NOT** run on every push. They only run when your commit message contains `[deploy]`.

```bash
# Normal push — no pipeline triggered
git commit -m "wip: working on something"
git push

# Triggers the pipeline
git commit -m "feat: update EKS node size [deploy]"
git push
```

| Event | Runs? |
|---|---|
| Push with `[deploy]` in message | ✅ Yes |
| Push without `[deploy]` | ❌ No |
| Pull Request (any commit) | ✅ Yes (plan only, no apply) |
| `workflow_dispatch` (manual) | ✅ Yes |
| Scheduled (drift detection, Monday) | ✅ Yes |

## Bootstrap in CI/CD

Bootstrap is the only step that requires a **one-time manual run**. After that, it's fully in CI/CD.

**Why one-time manual?** Bootstrap creates the GitHub Actions OIDC role. Without that role, GitHub Actions can't authenticate to AWS to run bootstrap. First time = manual. Every time after = CI/CD.

**How it works after first run:**
- Bootstrap state is stored in `s3://YOUR_BOOTSTRAP_BUCKET` (a pre-created seed bucket)
- When you change files in `bootstrap/` and push with `[deploy]`, CI/CD runs `terraform plan` → `terraform apply` for bootstrap automatically
- The seed bucket (`YOUR_BOOTSTRAP_BUCKET`) was created once with `aws s3 mb` — it is **not** managed by Terraform

## PHASE 3: Platform Bootstrap (run once after EKS is up)

This step installs ArgoCD and applies the App-of-Apps. After this, ArgoCD owns the cluster.

### 3.1 Run the Platform Bootstrap Workflow

1. Go to `techitfactory-infra` → Actions → **Platform Bootstrap**
2. Click **Run workflow**
3. Select environment: `dev`
4. GitOps repo: `YOUR_GITHUB_ORG/techitfactory-gitops`
5. Click **Run workflow**

The workflow will:
- Install ArgoCD via Helm on the EKS cluster
- Configure ArgoCD repo credentials (uses `GITOPS_TOKEN`)
- Apply `apps/root-app.yaml` → bootstraps App-of-Apps

### 3.2 Verify ArgoCD Is Running

```bash
kubectl get pods -n argocd
# All pods should be Running

# Get the initial admin password
kubectl -n argocd get secret argocd-initial-admin-secret \
  -o jsonpath="{.data.password}" | base64 -d
echo ""
```

### 3.3 Access ArgoCD UI

```bash
kubectl port-forward svc/argocd-server -n argocd 8080:443
# Open: https://localhost:8080
# Username: admin
# Password: from step 3.2
```

### 3.4 Verify Platform Apps Are Syncing

In the ArgoCD UI you should see:
- `root-app` → Synced ✅
- `nginx-ingress` → Syncing / Synced ✅
- `cluster-autoscaler` → Syncing / Synced ✅
- `metrics-server` → Syncing / Synced ✅

Or via CLI:
```bash
kubectl get pods -n ingress-nginx
# nginx-ingress-controller pods should appear

kubectl get pods -n kube-system | grep -E "autoscaler|metrics"
# cluster-autoscaler and metrics-server pods
```

### 3.5 Get the NLB Hostname (Public URL)

After NGINX Ingress is running, AWS creates an NLB:
```bash
kubectl get svc -n ingress-nginx
# EXTERNAL-IP column shows the NLB DNS name
# e.g.: abc123.elb.ap-south-1.amazonaws.com
```

This is your cluster's public endpoint. All HTTP traffic routes through here.

### 3.6 Create Kubernetes Secrets (REQUIRED before apps start)

The application pods reference a Kubernetes Secret called `app-secrets` for the JWT signing key.
This must be created manually — it cannot be stored in Git (it's sensitive).

```bash
# Create the app-secrets Secret in the techitfactory namespace
kubectl create secret generic app-secrets \
  --namespace techitfactory \
  --from-literal=jwt-secret="$(openssl rand -hex 32)"

# Verify it was created
kubectl get secret app-secrets -n techitfactory
```

> **Why this is manual:** Secrets must never be stored in Git. In production you would use
> AWS Secrets Manager + External Secrets Operator to sync them automatically. For now, this
> one-time manual step is required after every fresh cluster creation.

> **If you forget this step:** Pods will show `CreateContainerConfigError` with the event
> `Error: secret "app-secrets" not found`. Run the command above and pods will restart within 30s.

## Troubleshooting

### Terraform apply fails with "state locked"
```bash
cd techitfactory-infra/environments/dev
terraform force-unlock <LOCK_ID>
```

### ArgoCD app stuck in OutOfSync
```bash
argocd app sync <app-name>
# Or in UI: click "Sync" → "Synchronize"
```

### Pod in CreateContainerConfigError
```bash
kubectl describe pod <pod-name> -n techitfactory | grep -A5 "Events:"
# If you see: Error: secret "app-secrets" not found
# Fix: create the missing secret
kubectl create secret generic app-secrets \
  --namespace techitfactory \
  --from-literal=jwt-secret="$(openssl rand -hex 32)"
# Pods will self-heal within 30 seconds
```

### Pod in ImagePullBackOff
```bash
kubectl describe pod <pod-name> -n techitfactory
# Check: is the ECR image tag correct in kustomization.yaml?
# Check: does the IAM role have ECR pull permissions?
aws ecr get-login-password --region ap-south-1 | \
  docker login --username AWS --password-stdin \
  YOUR_AWS_ACCOUNT_ID.dkr.ecr.ap-south-1.amazonaws.com
```

### NGINX Ingress controller pending (NLB not created)
```bash
kubectl describe svc -n ingress-nginx ingress-nginx-controller
# Check Events for errors
# Verify EKS node IAM role has elasticloadbalancing:* permissions
```

### GitOps push conflict in CI
```bash
# ci.yml uses git pull --rebase before push — this handles most cases
# If it still fails, re-run the failed workflow job
```
