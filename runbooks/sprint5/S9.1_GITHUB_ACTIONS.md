## Part 1: Full GitOps Loop (App Pipeline)

```
Developer pushes to main
          ↓
detect-changes: identifies which of 6 services changed
          ↓
test: lint + tests for changed services only (parallel matrix)
          ↓
build: builds Docker image → pushes to ECR with commit SHA tag
       + Trivy security scan (CRITICAL/HIGH fails the build)
          ↓
update-gitops: uses yq to update environments/dev/<service>/kustomization.yaml
               newTag: <commit-sha>
               Commits to techitfactory-gitops (max-parallel: 1)
          ↓
ArgoCD auto-syncs within ~3 minutes → new pod running in cluster
```

### Change Detection
| Push touches | What builds |
|---|---|
| `services/api-gateway/**` | Only api-gateway |
| `services/product/**` | Only product-service |
| `services/order/**` | Only order-service |
| `services/cart/**` | Only cart-service |
| `services/user-service/**` | Only user-service |
| `services/frontend/**` | Only frontend |
| No services changed | Nothing builds |

## Part 2: Terraform Pipeline (Separate Repo)

The Terraform pipeline is **completely separate** from the app pipeline.
It lives in `techitfactory-infra` and is triggered only when Terraform files change.

### Key design decisions:
| Decision | Why |
|---|---|
| Path filters per environment | `modules/**` change triggers both dev+prod plans |
| `concurrency: cancel-in-progress: false` | Never cancel Terraform — a mid-run cancel leaves state locked |
| Plan saved as artifact, apply uses that plan | You review the exact plan in PR; apply doesn't re-plan |
| Prod apply requires `environment: production-infra` approval | Human must approve before prod infra changes |
| Drift detection (scheduled Monday 06:00 UTC) | Alerts if live infra diverges from Git |
| TF_VERSION pinned to `1.9.8` | Avoid surprise breakage from new Terraform releases |

### Terraform Job Flow
```
PR:
  detect-changes → tf-dev-plan   → PR comment
               └→ tf-prod-plan  → PR comment

Merge to main:
  detect-changes → tf-dev-plan  → tf-dev-apply  (auto)
               └→ tf-prod-plan → tf-prod-apply  (needs approval + waits for dev)

Scheduled (Monday):
  tf-dev-plan  → drift alert if changes exist
  tf-prod-plan → drift alert if changes exist
```

## Part 4: GitHub Environments to Configure

Two environments need to be created in GitHub (repo Settings → Environments):

### `production` (in techitfactory-app)
- **Used by:** `release.yml` → `update-prod-gitops` job
- **Protection rules:**
  - ✓ Required reviewers: add your team leads
  - ✓ Deployment branches: tags only (`v*`)
  - ✓ Wait timer: 5 minutes (optional)

### `production-infra` (in techitfactory-infra)
- **Used by:** `terraform-ci.yml` → `tf-prod-apply` job
- **Protection rules:**
  - ✓ Required reviewers: add your infra owners
  - ✓ Deployment branches: main only

## Part 5: Production Gates Summary

| Gate | Where | What It Blocks |
|---|---|---|
| Trivy scan (CRITICAL/HIGH) | ci.yml + release.yml | Images with known unfixed CVEs |
| SonarCloud quality gate | sonarcloud.yml | PRs with < 70% coverage, bugs, vulns |
| kustomize build validation | validate.yml | Broken GitOps manifests before ArgoCD sees them |
| Terraform plan review | terraform-ci.yml | Infra PRs without a reviewed plan |
| `production` environment approval | release.yml | Prod GitOps updates without a reviewer |
| `production-infra` env approval | terraform-ci.yml | Prod infra applies without a reviewer |
| Semver tag validation | release.yml | Releases with non-vX.Y.Z tags |

## Part 6: What Was Fixed From the Original

| Issue | Old Behavior | Fixed |
|---|---|---|
| Terraform CI covered only `bootstrap/` | Plan never ran for VPC/EKS/ECR changes | Path-filtered jobs per environment |
| No `terraform apply` | Manual apply always required | Auto-apply on merge (with approval for prod) |
| `sed` for GitOps update | Fragile — matched wrong YAML fields | `yq -i '.images[0].newTag'` — precise |
| No Trivy on release builds | Prod images shipped without security scan | Trivy runs in `build-all-services` |
| `sonarcloud.yml` had `continue-on-error: true` | Quality gate never blocked anything | Removed — gate hard-fails the build |
| `validate.yml` used `\|\| echo "Warning"` | Broken kustomize overlays passed | Strict fail on any build error |
| No `concurrency:` on app pipeline | Multiple CI runs raced for GitOps pushes | `concurrency: ci-${{ github.ref }}` |
| Slack in runbook but not in workflows | Release notifications never sent | `slackapi/slack-github-action@v1.27.0` added |
| `@master` pin on SonarCloud action | Breaking changes could arrive silently | Pinned to `@v3` |

## Verification Checklist

### App Pipeline
- [ ] Push a change to one service — only that service builds
- [ ] PR shows test results before merge is allowed
- [ ] Trivy scan runs and passes (or blocks on real CVEs)
- [ ] GitOps repo updated with new SHA tag (check `git log` in techitfactory-gitops)
- [ ] ArgoCD auto-syncs within 3 minutes
- [ ] `kubectl get pods -n techitfactory` shows updated image SHA

### Release Pipeline
- [ ] Create release with tag `v1.0.0`
- [ ] All 6 services build and pass Trivy
- [ ] Approval prompt appears for `production` environment
- [ ] After approval: prod GitOps updated with `v1.0.0` tag
- [ ] Slack notification sent

### Terraform Pipeline
- [ ] Open PR changing `environments/dev/main.tf` — plan comment appears
- [ ] Merge to main — `tf-dev-apply` runs automatically
- [ ] Open PR changing `environments/prod/` — `production-infra` approval gate appears

### GitOps Validation
- [ ] Introduce a YAML syntax error — validate.yml fails and blocks the PR
- [ ] Break a kustomization.yaml — kustomize build step fails
