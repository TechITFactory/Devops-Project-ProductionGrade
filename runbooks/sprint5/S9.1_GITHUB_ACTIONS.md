## Part 1: Workflow Architecture

### Full Pipeline Overview
```
techitfactory-infra/
└── .github/workflows/
    └── terraform-ci.yml        ← Already exists: plan on PR, no apply (manual)

techitfactory-app/
└── .github/workflows/
    ├── ci.yml                  ← Already exists: change detection + build + ECR push + GitOps update
    └── release.yml             ← Already exists: tagged release → prod GitOps update

techitfactory-gitops/
└── .github/workflows/
    └── validate.yml            ← Already exists: YAML validation + kustomize build check
```

### ci.yml — How It Works
```
Push to main with changes in services/*
         ↓
detect-changes: identifies which of 6 services changed
         ↓
test: runs lint + tests ONLY for changed services (parallel matrix)
         ↓
build: builds + pushes Docker image to ECR (ONLY on merge to main)
         ↓
update-gitops: commits new image SHA to environments/dev/<service>/kustomization.yaml
         ↓
ArgoCD auto-syncs the change to the cluster
```

### Change Detection Logic
| Change | What Builds |
|--------|-------------|
| `services/api-gateway/**` | Only api-gateway |
| `services/product/**` | Only product-service |
| `services/order/**` | Only order-service |
| `services/cart/**` | Only cart-service |
| `services/user-service/**` | Only user-service |
| `services/frontend/**` | Only frontend |
| No services changed | Nothing builds |

## Part 3: Setting Up AWS_ROLE_ARN

The OIDC role was created by the bootstrap Terraform. Get the ARN:

```bash
cd ~/Desktop/Devops-Project/techitfactory-infra/bootstrap
terraform output -raw github_actions_role_arn
# Output: arn:aws:iam::535002890483:role/techitfactory-github-terraform
```

Add this to **both** `techitfactory-infra` and `techitfactory-app` repos:
1. Go to GitHub repo → **Settings → Secrets and variables → Actions**
2. Click **New repository secret**
3. Name: `AWS_ROLE_ARN`
4. Value: `arn:aws:iam::535002890483:role/techitfactory-github-terraform`

> **Why the same role for both?** The bootstrap OIDC trust policy allows all three
> TechITFactory repos (`techitfactory-infra:*`, `techitfactory-app:*`,
> `techitfactory-gitops:*`) to assume this role. The role has both Terraform
> permissions AND ECR push permissions.

### Verify the Role Exists
```bash
aws iam get-role \
  --role-name techitfactory-github-terraform \
  --profile techitfactory \
  --query 'Role.Arn' --output text
```

## Part 4: Setting Up GITOPS_TOKEN

The `ci.yml` workflow pushes to `techitfactory-gitops` to update image tags. It
needs a GitHub Personal Access Token with write access to that repo.

### Create a Fine-Grained PAT (Recommended)

Fine-grained PATs are scoped to specific repos — safer than classic PATs.

1. Go to GitHub → **Settings → Developer settings**
2. **Personal access tokens → Fine-grained tokens → Generate new token**
3. Configure:
   - **Token name**: `techitfactory-gitops-write`
   - **Expiration**: 90 days (or custom)
   - **Resource owner**: `TechITFactory` (the org)
   - **Repository access**: `Only select repositories` → select `techitfactory-gitops`
   - **Permissions → Contents**: `Read and write`
4. Click **Generate token** and copy it immediately

### Add to techitfactory-app Secrets
1. Go to `techitfactory-app` repo → **Settings → Secrets and variables → Actions**
2. **New repository secret**
3. Name: `GITOPS_TOKEN`
4. Value: paste the token

> The workflow checks out `techitfactory-gitops` using this token, updates the
> `newTag` field in `kustomization.yaml`, then commits and pushes.

## Part 5: Enable GitHub OIDC Permissions in Each Repo

The CI workflow uses OIDC (`id-token: write`) to authenticate with AWS. This
is already in the workflow files, but verify the Actions settings allow it:

1. Go to each repo → **Settings → Actions → General**
2. Under **Workflow permissions**: select **Read and write permissions**
3. Check **Allow GitHub Actions to create and approve pull requests** (optional)

## Part 6: Test the CI Pipeline

### Trigger a CI Run
```bash
cd ~/Desktop/Devops-Project/techitfactory-app

# Make a trivial change to one service
echo "# CI test $(date)" >> services/api-gateway/README.md

git add services/api-gateway/README.md
git commit -m "test: trigger CI for api-gateway"
git push
```

Watch the run in GitHub → **Actions tab**

### Expected Workflow Steps
1. `detect-changes` → identifies `api-gateway: true`, builds matrix
2. `test` → runs `npm ci && npm run lint && npm test` for api-gateway only
3. `build` → builds Docker image, pushes to ECR with commit SHA tag, runs Trivy scan

```bash
  --repository-name techitfactory/api-gateway \

  --output table
```

```bash
# Should show a 40-char git SHA
```

```bash

# Status should show: Synced, Healthy, image tag matches the SHA
```

## Part 7: Troubleshooting Common CI Failures

### "Error: Credentials could not be loaded"
The OIDC role trust policy doesn't include your repo. Check:
```bash
aws iam get-role \
  --role-name techitfactory-github-terraform \
  --profile techitfactory \
  --query 'Role.AssumeRolePolicyDocument'
```
The `StringLike` condition should list `repo:TechITFactory/techitfactory-app:*`.

### "denied: Your authorization token has expired"
The ECR login step may have stale creds. This shouldn't happen with OIDC but if
it does, check that the role has `ecr:GetAuthorizationToken` permission.

### "GITOPS_TOKEN: Resource not accessible"
The fine-grained PAT may not have `Contents: write` on `techitfactory-gitops`.
Regenerate the token and check the permission setting.

### "No services changed" — workflow skips build
The changed file isn't in `services/**`. Check the paths-filter config in `ci.yml`:
```yaml
filters:
  api-gateway:
    - 'services/api-gateway/**'
```
