# Sprint 4, Story 8.3: Observability in Action

## Story Summary
Connect the running services to the monitoring stack and prove it works with
real data — not just placeholder dashboards.

**What we're verifying:**
- Prometheus is scraping all 6 services
- Real HTTP metrics flowing into Grafana
- Loki collecting and querying real application logs
- Alert rules firing correctly

---

## Prerequisites
- S8.1 completed (all pods Running)
- S7.1–S7.3 completed (Prometheus, Grafana, Loki installed)
- Port-forwards or access to Grafana UI

---

## Part 1: Verify Prometheus is Scraping All Services

```bash
# Port-forward Prometheus
kubectl port-forward svc/prometheus-kube-prometheus-prometheus -n monitoring 9090:9090
```

Open http://localhost:9090 → **Status → Targets**

You should see `techitfactory-apps` job with all 6 pod targets showing `UP`.

If any target is `DOWN` — check the pod has the scrape annotations:
```bash
kubectl get pods -n techitfactory -o jsonpath=\
  '{range .items[*]}{.metadata.name}{"\t"}{.metadata.annotations.prometheus\.io/scrape}{"\n"}{end}'
```

All pods should show `true`. If not, the base deployment is missing annotations.

### Run a Test Query in Prometheus

```bash
# Via curl (while port-forward is active)
curl -s 'http://localhost:9090/api/v1/query?query=up{job="techitfactory-apps"}' \
  | jq '.data.result[].metric.pod'
```

You should see all 6 service pod names listed.

## Part 2: Generate Real Traffic for Metrics

Before checking Grafana, generate some HTTP traffic so there's actual data to show:

```bash
# Port-forward API gateway in another terminal
kubectl port-forward svc/api-gateway -n techitfactory 8000:80 &

# Generate traffic — products, health, and some errors
for i in {1..50}; do
  curl -s http://localhost:8000/api/products > /dev/null
  curl -s http://localhost:8000/health > /dev/null
  curl -s http://localhost:8000/api/nonexistent > /dev/null   # 404s for error rate
done

echo "Traffic generated"
```

## Part 4: Loki — Query Real Logs

```bash
# Port-forward Loki directly
kubectl port-forward svc/loki -n monitoring 3100:3100
```

### Query Logs via CLI
```bash
# All logs from techitfactory namespace (last 5 minutes)
curl -G -s "http://localhost:3100/loki/api/v1/query_range" \
  --data-urlencode 'query={namespace="techitfactory"}' \
  --data-urlencode 'limit=20' \
  --data-urlencode "start=$(date -d '5 minutes ago' +%s)000000000" \
  --data-urlencode "end=$(date +%s)000000000" \
  | jq '.data.result[0].values[-1][1]'
```

### Query Logs in Grafana

1. Open Grafana → **Explore** (compass icon)
2. Switch data source to **Loki**
3. Run these LogQL queries:

```logql
# All app logs
{namespace="techitfactory"}

# Only error logs
{namespace="techitfactory"} |= "error"

# Logs from specific service
{namespace="techitfactory", app="api-gateway"}

# Error rate per service (metric query)
sum(rate({namespace="techitfactory"} |= "error" [5m])) by (app)
```

## Part 6: The Golden Signals

For every service in production, track these 4 signals:

| Signal | Prometheus Query | Meaning |
|--------|-----------------|---------|
| **Latency** | `histogram_quantile(0.99, rate(http_request_duration_seconds_bucket{namespace="techitfactory"}[5m]))` | P99 response time |
| **Traffic** | `sum(rate(http_requests_total{namespace="techitfactory"}[5m])) by (service)` | Requests/sec |
| **Errors** | `sum(rate(http_requests_total{namespace="techitfactory",status=~"5.."}[5m])) / sum(rate(http_requests_total{namespace="techitfactory"}[5m]))` | Error % |
| **Saturation** | `container_memory_usage_bytes{namespace="techitfactory"} / container_spec_memory_limit_bytes{namespace="techitfactory"}` | Memory pressure |

Run each query in Prometheus UI to verify data is flowing.
