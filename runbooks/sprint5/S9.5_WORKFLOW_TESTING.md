# Sprint 5 — Workflow Testing Guide

How to trigger and verify all 4 workflows in `techitfactory-app`.

---

## Overview

| # | Workflow | Trigger | Purpose |
|---|---|---|---|
| 1 | `ci.yml` | Push with `[deploy]` + files in `services/**` | Test → Build → GitOps update |
| 2 | `build-all.yml` | Manual `workflow_dispatch` | Rebuild all 6 services at once |
| 3 | `sonarcloud.yml` | Push/PR to files in `services/**` | Code quality gate |
| 4 | `release.yml` | GitHub Release published (tag `vX.Y.Z`) | Prod promotion |

---

## Workflow 1 — `ci.yml` (Main CI Loop)

### What it does
```
Push commit → Detect which services changed → Test → Build & push to ECR
→ Update kustomization.yaml in gitops repo → ArgoCD syncs → New pod live
```

### How to trigger
Make a real code change in any service directory:

```bash
cd /home/jai/Desktop/Devops-Project/techitfactory-app

# Make a visible frontend change (shows up in browser after deploy)
sed -i 's/Production-Grade Kubernetes Platform/Production-Grade Kubernetes Platform | Sprint 5 Live!/' \
  services/frontend/public/index.html

git add services/frontend/public/index.html
git commit -m "demo: sprint5 live CI/CD showcase [deploy]"
git push origin main
```

> **[deploy] is required.** Without it, pushes to `main` are ignored by the pipeline.
> PRs and `workflow_dispatch` always run regardless.

### What to watch (open 4 windows)

**Window 1 — GitHub Actions:**
```
github.com/YOUR_GITHUB_ORG/techitfactory-app → Actions → CI
```

**Window 2 — ArgoCD UI:**
```bash
kubectl port-forward svc/argocd-server -n argocd 8080:80
# Open http://localhost:8080
# Watch frontend app: Syncing → Synced
```

**Window 3 — Pods:**
```bash
watch kubectl get pods -n techitfactory -l app=frontend
```

**Window 4 — GitOps tag:**
```bash
watch -n5 "cd /home/jai/Desktop/Devops-Project/techitfactory-gitops && \
  git pull -q && grep newTag environments/dev/frontend/kustomization.yaml"
```

### Expected timeline
```
t=0:00  You pushed
t=0:20  GitHub Actions: CI starts, Detect Changes job runs
t=0:40  frontend = true detected
t=1:00  Test job starts → npm install → (no tests in frontend, skips)
t=1:30  Build job starts → docker build
t=3:00  Image pushed to ECR → Trivy scan
t=4:00  Update GitOps starts → yq updates kustomization.yaml → git push
t=5:00  Window 4: newTag changes to new SHA ← GitOps updated
t=5:30  Window 2: ArgoCD shows "Syncing..."
t=6:00  Window 3: new pod ContainerCreating, old pod Terminating
t=6:30  Window 3: new pod Running ← deployed!
t=7:00  Window 2: Synced ✓ Healthy
```

### Verify in browser
```
http://<your-NLB-hostname>
```
You should see "Production-Grade Kubernetes Platform | Sprint 5 Live!" in the subtitle.

Get NLB hostname:
```bash
kubectl get svc -n ingress-nginx nginx-ingress-ingress-nginx-controller \
  -o jsonpath='{.status.loadBalancer.ingress[0].hostname}'
echo ""
```

## Workflow 2 — `build-all.yml` (Seed All Services)

### What it does
Builds all 6 services in parallel, pushes to ECR, then updates all kustomization.yaml files
in the gitops repo sequentially. Use this after a fresh cluster — the normal `ci.yml` only
builds services that changed.

### How to trigger
```
GitHub → techitfactory-app → Actions → Build All Services → Run workflow

Inputs:
  environment: dev          ← which gitops env to update
  image_tag: (leave blank)  ← uses git SHA by default

→ Run workflow
```

### What to watch
```bash
# All 6 build jobs run in parallel
# Then 6 gitops update jobs run one at a time (sequential, to avoid git conflicts)
kubectl get pods -n techitfactory -w
# After gitops is updated, ArgoCD deploys all services
```

### Verify
```bash
# All 6 repos should have the same new SHA tag
for repo in frontend api-gateway user-service order-service product-service cart-service; do
  echo -n "$repo: "
  aws ecr describe-images \
    --repository-name techitfactory/$repo \
    --region ap-south-1 \
    --profile techitfactory \
    --query 'imageDetails | sort_by(@, &imagePushedAt) | [-1].imageTags[0]' \
    --output text
done
```

## Workflow 3 — `sonarcloud.yml` (Code Quality Gate)

### What it does
Runs on every push to files in `services/**`. Installs deps, runs tests with coverage,
sends results to SonarCloud. Quality Gate blocks merges if bugs/vulnerabilities found.

### Prerequisite
`SONAR_TOKEN` secret must be added to the repo. See `S9.2_SONARCLOUD.md`.

### How to trigger (automatic)
Any push that touches `services/**` automatically triggers SonarCloud.
The same push that triggers `ci.yml` will also trigger `sonarcloud.yml`.

### How to trigger manually
```
GitHub → techitfactory-app → Actions → SonarCloud Analysis → Run workflow
```
Or push any change to `services/**` or `sonar-project.properties`.

### Verify
```
github.com/YOUR_GITHUB_ORG/techitfactory-app → Actions → SonarCloud Analysis
→ Check the latest run passed
```

View results at: `sonarcloud.io/organizations/techitfactory`

## Workflow 4 — `release.yml` (Production Release)

### What it does
```
GitHub Release published with vX.Y.Z tag
→ Validate tag format
→ Build all 6 services (tagged with release version, not SHA)
→ Trivy scan every image
→ PAUSE for manual approval (GitHub "production" environment gate)
→ Update gitops/environments/prod/ kustomization files
→ Slack notification
→ Human manually runs: argocd app sync <service>-prod (final safety step)
```

### How to trigger

**Option A — GitHub UI (recommended for showcase):**
```
GitHub → techitfactory-app → Releases → "Draft a new release"
  → Choose a tag: v1.0.0  (type it, click "Create new tag: v1.0.0")
  → Release title: "Sprint 5 Release v1.0.0"
  → Description: "First production release — all 6 services"
  → Click "Publish release"
```

**Option B — CLI:**
```bash
cd /home/jai/Desktop/Devops-Project/techitfactory-app
git tag v1.0.0
git push origin v1.0.0

# Then create the GitHub Release from the tag:
gh release create v1.0.0 \
  --title "Sprint 5 Release v1.0.0" \
  --notes "First production release — all 6 services"
```

### Approval gate
After all 6 services build and pass Trivy, the `update-prod-gitops` job pauses:
```
GitHub → Actions → Release to Production → (click the run)
→ "update-prod-gitops" job shows "Waiting for review"
→ Click "Review deployments" → select "production" → "Approve and deploy"
```

### After approval
```bash
# ArgoCD prod apps use MANUAL sync — you must sync them explicitly
argocd app sync frontend-prod
argocd app sync api-gateway-prod
# ... (or sync all at once in ArgoCD UI)
```

### Tag format rules
Tag MUST match `vX.Y.Z` exactly:
```
v1.0.0   ✅ valid
v1.0     ❌ invalid (rejected by validate-release job)
1.0.0    ❌ invalid (missing "v" prefix)
v1.0.0-rc1 ❌ invalid (no pre-release suffixes)
```

## Common Issues

### `ci.yml` didn't trigger
- Check commit message has `[deploy]`
- Check the changed files are under `services/**` or `.github/workflows/ci.yml`
- PRs and `workflow_dispatch` bypass the `[deploy]` check

### `build-all.yml` Trivy fails
- Already fixed: uses apt repo install, not GitHub releases API
- If still failing: check if `sudo apt-get update` succeeded in logs

### `sonarcloud.yml` fails: `SONAR_TOKEN not found`
- Add the secret: `GitHub → techitfactory-app → Settings → Secrets → SONAR_TOKEN`
- Get the token from sonarcloud.io

### `release.yml` stuck at approval
- Go to `GitHub → Actions → (the release run) → Review deployments → Approve`
- Reviewer must be configured in `GitHub → Settings → Environments → production`

### Wrong service detected by `ci.yml`
The path filters in `detect-changes`:
```
api-gateway    → services/api-gateway/**
product-service → services/product/**
order-service  → services/order/**
cart-service   → services/cart/**
user-service   → services/user-service/**
frontend       → services/frontend/**
```
Change files in the matching directory to trigger that service's build.
