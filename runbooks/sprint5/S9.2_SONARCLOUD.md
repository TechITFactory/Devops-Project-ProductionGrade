# Sprint 5, Story 9.2: SonarCloud — Production-Grade Quality Gate

## Story Summary
Integrate SonarCloud so every PR has a real quality gate — not just a dashboard metric.
The gate **blocks merges** when coverage drops, bugs appear, or vulnerabilities are found.
The workflow already exists at `techitfactory-app/.github/workflows/sonarcloud.yml`.
This story is about configuring the SonarCloud side and setting up branch protection rules.

---

## What Was Fixed From the Original

| Old | Fixed |
|---|---|
| `npm ci \|\| true` — install failures hidden | Hard fail — no `\|\| true` anywhere |
| `npm test \|\| true` — test failures hidden | Hard fail — tests must pass |
| `continue-on-error: true` on SonarCloud scan | Removed — gate blocks the pipeline |
| `@master` pin on SonarCloud action | Pinned to `@v3` — no surprise breaking changes |
| Runs on all pushes | Path filters — only runs when `services/**` changes |

## Part 1: SonarCloud Account Setup

### Step 1.1: Create Account & Import Org
1. Go to [sonarcloud.io](https://sonarcloud.io)
2. Sign in with GitHub
3. **Analyze new project** → select `TechITFactory` org → select `techitfactory-app`
4. Choose **"With GitHub Actions"** (skip the auto-generated YAML — ours is already written)

### Step 1.2: Get the Project Token
1. SonarCloud → My Account → Security → **Generate Token**
2. Name it `techitfactory-app-ci`
3. Copy the token (shown only once)

### Step 1.3: Add Secret to GitHub
1. `techitfactory-app` → Settings → Secrets and variables → Actions
2. **New repository secret** → Name: `SONAR_TOKEN` → Value: paste token

## Part 2: Create sonar-project.properties

This file tells SonarCloud where to find sources and coverage reports.
Create it at the **root of `techitfactory-app/`**:

```properties
# sonar-project.properties
sonar.organization=techitfactory
sonar.projectKey=TechITFactory_techitfactory-app

# Monorepo — scan everything under services/
sonar.sources=services
sonar.exclusions=**/node_modules/**,**/dist/**,**/.next/**,**/__pycache__/**,**/*.test.*,**/*.spec.*

# Coverage reports written by the workflow's test step
sonar.javascript.lcov.reportPaths=services/api-gateway/coverage/lcov.info,services/product/coverage/lcov.info,services/cart/coverage/lcov.info,services/user-service/coverage/lcov.info
sonar.python.coverage.reportPaths=services/order/coverage.xml

# CRITICAL: Makes the SonarCloud GitHub Action block until gate passes/fails
# Without this, the step completes immediately and always shows green
sonar.qualitygate.wait=true
```

> **`sonar.qualitygate.wait=true` is the most important line.**
> Without it, the workflow step exits immediately and never fails — the gate is decorative.
> With it, the step blocks until SonarCloud processes the results and returns pass/fail.

## Part 3: How the Workflow Works

The workflow at `sonarcloud.yml` does this in order:

```
1. Checkout (full history — fetch-depth: 0)
         ↓
2. Install Node.js + Python deps (hard fail on error)
         ↓
3. Run tests WITH --coverage for all Node.js services
   Run pytest --cov for order-service (Python)
   (hard fail — bad tests block the gate)
         ↓
4. SonarCloud scan (pinned @v3)
   Reads coverage reports from services/*/coverage/
   sonar.qualitygate.wait=true → blocks until gate result
         ↓
5. Gate passes → PR can be merged
   Gate fails → PR is blocked
```

## Part 4: Configure the Quality Gate in SonarCloud

### Create a Custom Gate (Recommended)
1. SonarCloud → Your Org → Quality Gates → **Create**
2. Name: `TechITFactory`
3. Set as default for your org

### Recommended Conditions for This Project
| Metric | Condition | Value |
|---|---|---|
| Coverage on New Code | is less than | 70% |
| Bugs | is greater than | 0 |
| Vulnerabilities | is greater than | 0 |
| Security Hotspots Reviewed | is less than | 100% |
| Duplicated Lines (%) on New Code | is greater than | 3% |

> These apply to **new code only** — existing code won't immediately block you.
> Start at 70% coverage; raise it as test coverage improves.

## Part 5: Enable PR Decoration (Gate Comment on PRs)

SonarCloud automatically comments on PRs when `GITHUB_TOKEN` is provided.
The workflow already passes it:

```yaml
env:
  GITHUB_TOKEN: ${{ secrets.GITHUB_TOKEN }}
  SONAR_TOKEN: ${{ secrets.SONAR_TOKEN }}
```

Enable in SonarCloud:
1. Your project → Administration → General Settings → **Pull Requests**
2. Enable: "Decorate pull requests"

### What the PR comment looks like
```
Quality Gate passed ✅

Coverage: 72.4% (+4.1% since last analysis)
Duplications: 1.1%
Bugs: 0  Vulnerabilities: 0  Code Smells: 2 (minor)
```

---

## Part 6: Block Merges Until Gate Passes (Branch Protection)

The gate only matters if it can block merges. Set this up:

1. `techitfactory-app` → Settings → Branches → **Add rule** for `main`
2. Check **"Require status checks to pass before merging"**
3. Search for and add: `SonarCloud Quality Gate`
4. Check **"Require branches to be up to date before merging"**

Now no PR can be merged if SonarCloud fails.
